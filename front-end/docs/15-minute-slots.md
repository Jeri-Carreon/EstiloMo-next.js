# 15-minute Slots — How They Work in EstiloMo

## Quick answer

- False: there is no _hard-coded_ immutable 15-minute buffer. The system defaults to 15-minute candidate intervals, but this is configurable.

## Where the default lives

- Default constant: `DEFAULT_CANDIDATE_INTERVAL_MINUTES = 15` in [front-end/src/lib/appointmentAvailability.ts](front-end/src/lib/appointmentAvailability.ts).

## How 15-minute (or other) increments are created

- The function `resolveCandidateIntervalMinutes(...)` reads an explicit `candidateIntervalMinutes` parameter or the `APPOINTMENT_START_INTERVAL_MINUTES` environment variable. If neither is valid, it falls back to the default (15 minutes).
- `generateAvailableTimes(...)` calls `resolveCandidateIntervalMinutes(...)` to get `intervalMinutes`, then iterates from `workingStartMinutes` up to `workingEndMinutes` in steps of `intervalMinutes`:

  for (let start = workingStartMinutes; start + serviceDurationMinutes <= workingEndMinutes; start += intervalMinutes) {
  // treat `start` as a candidate appointment start time
  }

- For each candidate `start`, the system checks:
  - if the `serviceDurationMinutes` fits before `workingEndMinutes`; and
  - if the candidate overlaps any `busyRanges` (existing appointments, breaks, blocked ranges); and
  - an optional `isStartAllowed(start, end)` predicate (for example, enforcing booking cutoff times).

- All of the above logic is implemented in [front-end/src/lib/appointmentAvailability.ts](front-end/src/lib/appointmentAvailability.ts) — see `resolveCandidateIntervalMinutes`, `generateAvailableTimes`, and `getDynamicAvailabilityForDate`.

## Purpose of the interval (why it exists)

- To present consistent, predictable candidate start times to users (e.g., 9:00, 9:15, 9:30).
- To make availability generation efficient by stepping in fixed increments instead of checking every minute.
- To align appointment start times with business rules and UI expectations (and to simplify blocking/overlap checks).

## How to change it

- Per-request: pass `candidateIntervalMinutes` into `generateAvailableTimes(...)` or `getDynamicAvailabilityForDate(...)`.
- Globally: set the environment variable `APPOINTMENT_START_INTERVAL_MINUTES` (the code caps parsed values reasonably).

## Notes and gotchas

- The system enforces `serviceDurationMinutes` when determining candidates — a very short service will still only be offered at candidate starts derived from the interval.
- Busy ranges are normalized and merged so overlapping appointments/breaks are handled correctly before candidate generation.
- The default is a sensible trade-off but not a hard requirement — you can change it if you need 5-, 10-, or 30-minute increments.
